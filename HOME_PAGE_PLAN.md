# FIFA 首页规划（实现约束版）

> 说明：本文件用于定义 FIFA 首页的信息架构、Tab 分工、布局规则与交互边界。后续首页及相关详情页开发，应以本文件为准执行。
>
> 当前首页规划不只是概念草图，而是进入实现阶段前的结构规范文档。后续页面开发需要尽量严格对齐参考截图中的布局关系、模块位置、信息顺序和显示规则。

## 1. 首页目标

- 建立一个围绕赛事浏览与玩法选择的首页结构。
- 同时支持 `比赛` 与 `玩法` 两类浏览入口。
- 保持当前项目既有的深色主题风格与控制台式信息密度。
- 为后续详情页和下单流程预留清晰的结构延展空间。

## 1.1 全局尺寸规则

- 字号、按钮高度、输入框尺寸、卡片内边距与模块间距，统一以 `移动端优先` 作为基线设计。
- 默认样式应先确保在小屏手机宽度下成立，再逐步向平板与桌面端放大。
- 不允许先按桌面端尺寸设计，再被动压缩到移动端。
- 所有新增页面、详情页、弹窗、抽屉与悬浮下单区，都必须遵循该规则。

## 2. 首页一级结构

首页由以下四个一级区域组成：

1. `Banner 区域`
2. `Tab 区域`
3. `内容主体区域`
4. `悬浮下单区域（按页面规则显示）`

其中：

- `Banner 区域` 固定在首页顶部。
- `Tab 区域` 位于 Banner 下方，仅包含两个一级 Tab：
  - `比赛`
  - `玩法`
- `内容主体区域` 会根据当前 Tab 不同而切换不同布局。
- `悬浮下单区域` 不是全局固定显示，而是按当前页面类型决定是否展示。

## 3. Tab 定义

### 3.1 比赛 Tab

- `比赛` Tab 对应“赛事列表页”。
- 布局采用 `左侧赛事列表 + 右侧悬浮下单区` 的双栏结构。
- 该页面的布局参考来源于第一张赛事列表截图。
- 该 Tab 的重点是：
  - 浏览按日期分组的比赛
  - 查看不同玩法列
  - 在同一页面配合右侧下单区完成操作

### 3.2 玩法 Tab

- `玩法` Tab 对应“玩法/盘口聚合页”。
- 布局采用 `全宽内容区`，不显示右侧悬浮下单区。
- 该页面的布局参考来源于第二张玩法卡片截图。
- 该 Tab 的重点是：
  - 浏览多个玩法卡片
  - 从聚合列表中进入某个玩法详情
  - 首页阶段不直接展示侧边下单区

## 4. 悬浮下单区显示规则

悬浮下单区不是所有页面都展示，规则如下：

### 4.1 首页比赛 Tab

- `1024px` 及以上：显示右侧悬浮下单区，进入 PC 双栏布局。
- `1024px` 以下：不显示固定侧栏，改为弹窗、抽屉或底部唤起面板。

### 4.2 首页玩法 Tab

- `1024px` 及以上：首页不显示悬浮下单区。
- `1024px` 以下：首页同样不显示固定下单区。
- 该页面内容区应占满可用宽度。

### 4.3 玩法详情页

- 用户点击玩法卡片进入详情页后，显示悬浮下单区。
- `1024px` 及以上可采用 `详情内容 + 右侧悬浮下单区` 布局。
- `1024px` 以下继续使用弹窗、抽屉或底部唤起式交互。

### 4.4 首页响应式断点规则

- `< 640px`：
  - 移动端基线尺寸，所有默认字号、间距、按钮高度优先按这一档设计。
- `640px - 1023px`：
  - 大手机与平板过渡区，仅做轻量放大，不进入固定右栏布局。
- `1024px` 及以上：
  - 进入 PC 布局规则。
  - 比赛页显示右侧固定下单区。
  - 移动端抽屉式下单区关闭。

### 4.5 悬浮下单区的组件化要求

- 悬浮下单区不会只在首页出现。
- 后续它还会在其他页面或详情页中复用。
- 因此悬浮下单区必须设计为独立组件，而不是写死在某一个页面内部。
- 该组件需要支持根据当前选中项动态切换展示内容。
- 组件设计时应优先考虑复用性、联动性与状态同步能力。

### 4.6 悬浮下单区的全局状态要求

- 悬浮下单区展示的数据不是静态文案，而是需要跟随页面中的选中项实时联动。
- 队伍信息、盘口信息、价格信息、金额信息等，都应由统一状态源驱动。
- 由于悬浮下单组件会在多个页面复用，因此相关数据不应只保存在局部页面状态中。
- 需要使用全局状态管理来维护当前下单上下文。
- 后续实现时，应保证以下信息可以全局同步：
  - 当前来源页面或模块
  - 当前选中的比赛或玩法
  - 当前选中的候选项
  - 当前盘口类型
  - 当前价格
  - 当前输入金额
  - 当前下单卡片是否展开或显示

### 4.7 悬浮下单区与列表的联动规则

- 当用户在比赛列表或玩法列表中点击某个可下注项时，悬浮下单区需要同步更新。
- 同步更新内容至少包括：
  - 标题信息
  - 队伍或对象名称
  - 对应价格
  - 盘口类型
  - 当前金额输入上下文
- 如果切换到新的选中项，悬浮下单区应刷新为新数据，而不是保留旧的展示内容。
- 如果没有选中项，应显示空状态或默认提示态。

## 5. 比赛 Tab 布局规范

### 5.1 页面整体布局

- 顶部为 `Banner`。
- Banner 下方为 `比赛 / 玩法` Tab 切换栏。
- 切换到 `比赛` 后：
  - `1024px` 及以上采用双栏布局
  - 左侧：赛事列表区域
  - 右侧：悬浮下单区域
  - `1024px` 以下采用移动端单列卡片布局

### 5.2 赛事列表组织方式

- 赛事列表按日期分组展示。
- 每个日期分组包含：
  - 日期标题
  - 多场比赛卡片
- 日期分组应具备清晰的上下层级与留白区分。
- 移动端首页卡片只展示 `胜负线`。
- `让分` 与 `总分` 不在移动端首页直接展开，而是在后续详情页中展示。

示例结构：

1. `Thu, June 11`
2. `Fri, June 12`
3. `Sat, June 13`
4. `Sun, June 14`

### 5.3 单场比赛卡片结构

每张比赛卡片需要遵循截图中的横向排布逻辑，主要分为三部分：

1. 左侧基础信息区
2. 中部/右侧玩法列区
3. 右上辅助操作区

#### 左侧基础信息区

- 比赛时间
- 交易量
- 主队信息
- 客队信息
- 队伍国旗/图标
- 当前比分

#### 中部/右侧玩法列区

- 采用按列展示的玩法信息结构。
- 当前至少包含三列：
  - `胜负线`
  - `让分`
  - `总分`
- 每列内部展示对应赔率项或可点击选项。
- 选项块需要保持规则栅格感和统一高度。

#### 右上辅助操作区

- 卡片右上角包含 `比赛视图` 按钮。
- 按钮旁可带数字徽标或统计标记。

### 5.4 比赛 Tab 的实现约束

- 必须保持“左侧列表、右侧下单区”的信息结构。
- 必须保持“按日期分组”的列表组织方式。
- 必须保持“比赛信息在左、玩法列在右”的卡片排布。
- 移动端首页必须改为单列大卡片结构，优先贴近参考图中的 `时间 + 交易量 + 两支队伍 + 胜负线三按钮`。
- 移动端首页不显示 `让分`、`总分` 和对应 `line selector`。
- `让分`、`总分` 与 line 选择器延后到详情页再展示。
- 不应将比赛列表改造成纯卡片宫格或瀑布流。
- 不应在首页比赛 Tab 中隐藏悬浮下单区。

## 6. 玩法 Tab 布局规范

### 6.1 页面整体布局

- 顶部保留 `Banner`。
- Banner 下方为 `比赛 / 玩法` Tab 切换栏。
- 切换到 `玩法` 后，主体区域占满全宽。
- 该页面不显示右侧悬浮下单区。

### 6.2 内容组织方式

- 页面主体为多列卡片网格布局。
- 每张卡片代表一个玩法主题、预测问题或盘口入口。
- 整体视觉更接近“市场聚合列表”，不是“按日期排列的赛事清单”。

### 6.3 玩法卡片基础结构

每张标准卡片包含以下信息模块：

- 左上图标、赛事图标或封面图
- 玩法标题
- 两个或多个候选项
- 各候选项对应的百分比/概率
- 对应的 `是 / 否` 操作按钮
- 底部交易量
- 收藏、标记或附加操作入口

### 6.4 玩法卡片的两种信息模板

`玩法` Tab 中并不是所有卡片都使用同一种信息结构。根据玩法类型不同，首页卡片需要分为两类展示模板：

#### 模板 A：多候选项列表卡片

适用场景：

- 一个玩法下存在多个候选对象。
- 例如：
  - 世界杯冠军
  - 小组冠军
  - 某类需要在多个队伍或对象之间进行比较的市场

展示目的：

- 强调多个候选对象之间的横向比较关系。
- 让用户快速看到当前最重要的候选项及其对应概率/价格。

结构特征：

- 卡片顶部展示玩法标题。
- 内容区使用候选项列表。
- 每一行包含：
  - 候选对象名称
  - 当前百分比/概率
  - 对应 `是`
  - 对应 `否`

首页展示限制：

- 当一个玩法下存在多个候选对象时，首页卡片中最多只展示 `2` 条候选数据。
- 这样做的目的，是为了保证首页卡片高度基本一致，避免因为列表项数量不同导致网格高度参差不齐。
- 剩余候选项不在首页卡片中继续展开，而是在详情页中完整展示。

实现约束：

- 首页不得因为候选项过多而无限向下扩展卡片高度。
- 首页卡片只承担“预览主要候选项”的职责。
- 完整候选列表应放到详情页承接。

#### 模板 B：单命题概率强化卡片

适用场景：

- 一个玩法本质上是单个问题的二元判断。
- 例如：
  - 某球员会不会参加世界杯
  - 某个事件会不会发生
  - 某个明确命题是否成立

展示目的：

- 强调一个核心命题当前的市场倾向。
- 用更强的视觉方式突出概率结果，而不是突出候选对象比较。

结构特征：

- 卡片主体围绕一个问题展开。
- 使用环形进度、圆弧百分比或大号概率数字展示当前倾向。
- 配置更强调的 `是 / 否` 操作按钮。
- 交易量与辅助操作仍保留在卡片底部。

实现约束：

- 这种卡片不需要展示候选对象列表。
- 其核心是“单问题 + 强化概率表达 + 直接操作”。

### 6.4 特殊卡片样式

- 页面中允许出现强化样式卡片。
- 例如：
  - 带人物头像的玩法卡片
  - 带圆弧进度/概率环的卡片
  - 更强调主问题与操作按钮的横向组合卡片
- 这些特殊卡片仍需保持与整体主题一致，不应脱离当前视觉系统。

### 6.5 页面底部扩展入口

- 页面底部预留 `显示更多盘口` 一类的加载或展开入口。
- 可作为分页、增量加载或跳转更多列表的入口。

### 6.6 玩法 Tab 的实现约束

- 必须使用满宽内容布局，不显示首页右侧悬浮下单区。
- 必须使用多列卡片网格，不改成按日期分组列表。
- 必须保留“玩法卡片聚合浏览”的页面定位。
- 多候选项卡片在首页最多展示 `2` 条候选项，不能因为数据更多而破坏卡片高度一致性。
- 超出首页预览范围的候选项，应进入详情页后完整展示。
- 点击卡片后进入详情页，再显示悬浮下单区。

## 7. 详情页延展规则

后续需要新增详情页时，建议按以下原则执行：

- `比赛详情页`：
  - 可延续比赛页结构，继续显示悬浮下单区。
- `玩法详情页`：
  - 详情内容为主，右侧显示悬浮下单区。
- 首页承担的是“聚合浏览入口”职责。
- 详情页承担的是“信息展开 + 下单操作”职责。
- 详情页的具体入口规则、默认选中规则与内容结构，应以 [DETAIL_PAGE_PLAN.md](/Users/daxiong/Desktop/workSpace/FIFA/DETAIL_PAGE_PLAN.md:1) 为准。

### 7.1 首页进入详情页的规则补充

- 比赛列表中，每场比赛卡片的 `非按钮区域` 可进入比赛详情页。
- 玩法卡片中，`非按钮区域` 可进入玩法详情页。
- 玩法卡片若存在多候选项：
  - 点击某一候选项所在行进入详情页时，应把该候选项作为预选结果带入详情页。
  - 点击卡片其他非候选项区域进入详情页时，不带显式候选项来源。

### 7.2 首页比赛区默认选中规则

- 首页 `比赛` Tab 若存在比赛数据，桌面端右侧悬浮下单区默认选中第一场比赛的第一个可交易项。
- 当前约定的默认项为该比赛 `胜负线` 的第一项。
- 移动端首页不因为该默认选中而自动弹出下单抽屉。

## 8. 悬浮下单卡片规范

本节基于当前下单区截图与业务裁剪要求整理，用于定义项目实际需要的下单卡片能力。

### 8.1 参考截图的使用原则

- 当前截图可作为悬浮下单区的视觉和信息层级参考。
- 但项目不需要照搬截图中的所有交互项。
- 需要基于业务实际做减法，保留核心必要能力。

### 8.2 本项目需要保留的能力

- 顶部基础信息区
- 当前选中对象或队伍信息
- 当前盘口类型展示
- 当前价格展示
- 金额输入区
- 快捷加额按钮
- 提交交易按钮

### 8.3 本项目需要裁剪掉的能力

- `买入 / 卖出` 双切换不需要完整保留。
- 项目中只保留 `买` 的能力。
- 下单区中不再额外展示“买入类型 / 买”这一段文案。
- 盘口按钮中不再额外展示 `BUY` 文案。
- 右侧盘口类型不需要做成下拉选择。
- 盘口类型只做当前选中内容的展示，不提供该区域内再次切换。

### 8.4 下单卡片的结构定义

结合当前业务要求，悬浮下单卡片建议包含以下模块：

1. 顶部信息区
2. 交易类型展示区
3. 当前选项展示区
4. 金额输入区
5. 快捷加额区
6. 提交操作区

#### 顶部信息区

- 展示当前比赛、玩法或问题标题。
- 展示当前选中的队伍、对象或候选项名称。
- 如果有图标、国旗、头像，可一并展示。

#### 交易类型展示区

- 仅保留 `买`。
- 不需要提供 `卖` 选项。
- 如果视觉上仍保留该区域，也应表现为单一有效类型，而非双 Tab 切换。

#### 当前选项展示区

- 展示当前被选中的下注项及其价格。
- 如果页面存在 `Yes / No`、队伍、胜负线或其他盘口选项，该区域展示当前生效项。
- 该信息必须由外部选择动作驱动同步更新。

#### 金额输入区

- 展示当前输入金额。
- 支持手动输入。
- 支持在不同选项切换后保留或重置金额，具体以后续交互规则为准。
- 至少当前文档层面，需要保证金额状态可被统一管理。
- 输入框不建议长期使用固定像素宽度。
- 应优先采用响应式宽度策略，例如最小宽度 + 相对宽度 + 最大宽度的组合。
- 在移动端和抽屉面板中，输入框需要避免与金额展示区互相挤压。

#### 快捷加额区

- 保留如 `+1`、`+5`、`+10`、`+100` 这类快捷加额按钮。
- 用于快速叠加输入金额。

#### 提交操作区

- 保留主操作按钮，例如 `交易`。
- 该按钮文案与状态后续可根据业务再细化。

### 8.5 下单卡片的联动数据要求

悬浮下单卡片中的以下信息，需要根据用户在页面中点击的列表项实时同步：

- 当前比赛标题
- 当前队伍或对象名称
- 当前盘口类型
- 当前价格
- 当前所选下注项
- 当前金额

这意味着：

- 列表项点击不是单纯高亮，而是要驱动下单卡片状态更新。
- 下单卡片本身不是独立静态组件，而是一个由全局状态驱动的业务组件。

### 8.6 下单卡片的复用要求

- 该卡片会在多个场景展示，不仅是首页。
- 因此不能把业务逻辑耦合在首页页面文件中。
- 需要拆分为可复用组件，例如：
  - `OrderPanel`
  - `OrderSelectionSummary`
  - `OrderAmountInput`
- 页面负责触发选中行为，组件负责消费全局状态并渲染。

### 8.7 状态管理建议

为了支撑跨页面复用和联动，下单卡片相关数据建议纳入全局状态管理。

建议至少维护以下状态：

- 当前下单上下文类型
  - 比赛
  - 玩法
- 当前选中实体 ID
- 当前选中标题
- 当前选中副标题或对象名称
- 当前盘口类型
- 当前价格
- 当前买入选项
- 当前输入金额
- 当前下单面板显示状态

建议的行为能力包括：

- 设置当前选中项
- 更新盘口类型
- 更新价格
- 更新金额
- 清空当前下单上下文
- 控制下单面板显示与隐藏

## 9. 视觉风格约束

首页不是单一内容流，而是一个带双入口的首页结构：
