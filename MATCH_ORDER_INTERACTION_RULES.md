# FIFA 比赛页交互规则

> 说明：本文件用于定义比赛列表、盘口切换、line 选择器、悬浮下单区之间的联动规则。后续比赛页与比赛详情页开发，应优先遵循本文件。

## 1. 交互目标

- 用户先进入某场比赛上下文。
- 再从不同市场类型中选择一个可交易盘口。
- 若该市场存在 `line` 维度，需要先确定当前 line 版本。
- 悬浮下单区始终展示“当前比赛 + 当前市场 + 当前 line + 当前方向”的最终结果。

## 2. 基本状态模型

比赛页的交互至少由以下几层状态组成：

1. `selectedMatch`
2. `selectedMarketType`
3. `selectedLineVariant`
4. `selectedOutcome`
5. `selectedTradeSide`

含义如下：

- `selectedMatch`
  - 当前选中的比赛，例如 `Mexico vs South Africa`
- `selectedMarketType`
  - 当前市场类型，例如：
    - `胜负线`
    - `让分`
    - `总分`
- `selectedLineVariant`
  - 当前市场对应的 line 版本
  - 仅在 `让分` 与 `总分` 市场中存在
- `selectedOutcome`
  - 当前用户选中的具体方向或结果
- `selectedTradeSide`
  - 当前下单区中实际选中的交易按钮
  - 在 `胜负线` 市场中通常表现为 `Yes / No`
  - 在 `让分` 与 `总分` 中通常表现为成对盘口中的某一边

## 3. 三类市场的交互模板

比赛列表中的盘口并不是同一种交互模型，需要按市场类型拆成三种模板。

### 3.1 胜负线

适用特征：

- 不依赖 line 选择。
- 结果本身就是明确候选项，例如：
  - `MEX`
  - `DRAW`
  - `RSA`

交互逻辑：

1. 用户点击某个胜负线候选项。
2. 页面进入该候选项对应的命题上下文。
3. 悬浮下单区展示：
   - 当前比赛
   - 当前候选对象
   - `Yes / No` 两个交易按钮
4. `Yes` 表示买入“该命题成立”。
5. `No` 表示买入“该命题不成立”。

示例：

- 点击 `MEX 67¢`
- 悬浮下单区变为：
  - 标题：`Mexico vs South Africa`
  - 对象：`Mexico`
  - 交易按钮：
    - `Yes 67¢`
    - `No 34¢`

规则结论：

- `胜负线` 使用 `命题式 Yes / No 模板`
- 不展示 line 选择器

### 3.2 让分

适用特征：

- 盘口不仅取决于球队，还取决于让分线。
- 同一个比赛会存在多个让分版本。

交互逻辑：

1. 用户点击 `让分` 列中的任一盘口。
2. 页面进入 `让分` 市场上下文。
3. 当前比赛卡片下方展开 `line selector`。
4. 用户可以切换不同的让分版本。
5. 每个让分版本对应一对成对盘口：
   - 一边是主队盘口
   - 一边是客队盘口
6. 悬浮下单区展示当前 line 下的这对盘口，并高亮当前选中的一边。

规则结论：

- `让分` 使用 `双边盘口模板`
- 必须展示 line 选择器
- 当前 line 切换后，盘口对与下单区同步更新

### 3.3 总分

适用特征：

- 市场围绕总进球数或总得分线展开。
- 当前方向固定是一对：
  - `Over`
  - `Under`

交互逻辑：

1. 用户点击 `总分` 列中的任一盘口。
2. 页面进入 `总分` 市场上下文。
3. 当前比赛卡片下方展开 `line selector`。
4. 用户可以切换不同总分线，例如：
   - `0.5`
   - `1.5`
   - `2.5`
   - `3.5`
5. 每条总分线都对应一对盘口：
   - `O x.x`
   - `U x.x`
6. 悬浮下单区展示当前 line 下的 `Over / Under` 成对盘口。

规则结论：

- `总分` 使用 `Over / Under 双边盘口模板`
- 必须展示 line 选择器
- 切换 line 后，下单区同步更新

## 4. line selector 规则

### 4.1 何时出现

- 仅在以下市场中出现：
  - `让分`
  - `总分`
- `胜负线` 不出现

### 4.2 让分的 line selector

让分的 line selector 与总分不同，它不是单纯的一组数值，而是 `line magnitude + 让分方向` 的组合。

这意味着：

- 同样是 `1.5`
- 可能存在两个不同的盘口版本：
  - 主队 `-1.5` / 客队 `+1.5`
  - 主队 `+1.5` / 客队 `-1.5`

因此：

- `让分` 的 line selector 可以出现重复的 `1.5`、`2.5`
- 因为这些重复值并不代表同一个盘口版本
- 它们只是数值相同，但让分方向不同

### 4.3 总分的 line selector

总分的 line selector 是单纯的数值线：

- `0.5`
- `1.5`
- `2.5`
- `3.5`
- `4.5`

它不区分主客队方向，因此不会重复。

## 5. 列表区域与下单区的联动

### 5.0 首页默认选中规则

- 首页 `比赛` Tab 只要存在比赛数据，桌面端右侧悬浮下单区应默认选中第一场比赛的第一个可交易项。
- 当前默认项定义为该比赛 `胜负线` 的第一项。
- 该默认选中只用于桌面端固定下单区的初始态。
- 移动端首页不因该默认选中自动弹出下单抽屉。

### 5.1 移动端首页限制

- 移动端首页比赛卡片只展示 `胜负线`。
- `让分` 与 `总分` 不在移动端首页列表直接出现。
- `让分`、`总分` 及其 `line selector` 延后到比赛详情页中展示。
- 因此移动端首页的主要操作是：
  - 浏览比赛
  - 直接点击胜负线
  - 通过抽屉式下单区完成胜负线下单

### 5.2 胜负线联动

点击候选项后：

- 比赛上下文同步到下单区
- 候选对象同步到下单区
- 当前价格同步到下单区
- 下单区切换为 `Yes / No` 模板

### 5.3 让分联动

点击让分列后：

- 当前比赛进入 `让分` 交互模式
- 展开 line selector
- 当前 line 版本写入全局状态
- 当前选中方向写入全局状态
- 下单区切换为 `双边盘口模板`

若用户再切换 line：

- 盘口对需要同步刷新
- 当前选中对象尽量保持语义一致
- 即保持“主队侧”或“客队侧”的连续性，而不是无条件重置

### 5.4 总分联动

点击总分列后：

- 当前比赛进入 `总分` 交互模式
- 展开 line selector
- 当前总分线写入全局状态
- 当前选中方向写入全局状态
- 下单区切换为 `Over / Under 双边盘口模板`

若用户再切换总分线：

- `Over / Under` 对应价格需要同步刷新
- 当前方向尽量保持延续，例如：
  - 已选 `Over`
  - 切换 line 后仍默认保持 `Over`
  - 直到用户主动改点 `Under`

## 6. 悬浮下单区模板规则

### 6.0 通用显示约束

- 下单区只保留真实需要的操作信息。
- 不再额外展示“买入类型 / 买”文案区块。
- 盘口按钮中不再额外展示 `BUY` 文案。
- 右上角市场类型标签应使用更轻量的小标签样式，不应抢占主标题视觉层级。
- 金额输入框不建议使用长期固定宽度，应采用响应式宽度策略。

### 6.1 胜负线模板

结构：

1. 比赛标题
2. 当前候选对象
3. `Yes / No` 按钮
4. 金额输入
5. 快捷加额
6. 提交按钮

标题示例：

- `Mexico vs South Africa`
- `Mexico`

按钮示例：

- `Yes 67¢`
- `No 34¢`

### 6.2 让分模板

结构：

1. 比赛标题
2. 当前选中方向，例如 `South Africa -1.5`
3. 当前 line 下的双边盘口按钮
4. 金额输入
5. 快捷加额
6. 提交按钮

标题示例：

- `Mexico vs South Africa`
- `South Africa -1.5`

按钮示例：

- `RSA -1.5 16.8¢`
- `MEX +1.5 97.7¢`

### 6.3 总分模板

结构：

1. 市场标题
2. 当前选中方向，例如 `Over 2.5`
3. 当前总分线下的双边按钮
4. 金额输入
5. 快捷加额
6. 提交按钮

标题示例：

- `Over vs Under`
- `Over 2.5`

按钮示例：

- `O 2.5 46¢`
- `U 2.5 55¢`

## 7. 项目实现要求

### 7.0 响应式断点要求

- `< 640px`：
  - 作为移动端默认样式基线。
- `640px - 1023px`：
  - 作为过渡区，仅做轻量放大。
- `1024px` 及以上：
  - 进入 PC 布局。
  - 显示右侧固定下单区。
  - 关闭移动端抽屉式下单交互。

### 7.1 数据结构要求

比赛数据不能只存一组扁平按钮文案。

至少要拆出：

- `winnerMarket`
- `spreadMarket`
- `totalMarket`

其中：

- `winnerMarket` 直接存候选项
- `spreadMarket` 需要存多个 line variant
- `totalMarket` 需要存多个 total line

### 7.2 状态管理要求

这些状态需要纳入全局管理：

- 当前选中比赛
- 当前选中市场类型
- 当前选中 line variant
- 当前选中方向
- 当前下单区展示状态
- 当前金额

### 7.3 组件职责要求

建议按以下职责拆分：

- `MatchList`
  - 负责渲染比赛列表与盘口入口
- `LineSelector`
  - 负责渲染让分/总分线值选择器
- `OrderPanel`
  - 负责根据当前全局状态切换三种模板

## 8. 当前结论

这套比赛页交互不是单层点击模型，而是：

1. 先进入比赛上下文
2. 再进入具体市场类型
3. 如果有 line，则先确定当前 line 版本
4. 最后确定当前交易方向
5. 悬浮下单区始终展示这个最终结果

可以用一句话概括：

`比赛决定上下文，市场决定模板，line 决定盘口版本，方向决定最终下单对象。`
